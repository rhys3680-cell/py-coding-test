### 필드 설명

| 필드 | 설명 |
|---|---|
| `date` | YYYY-MM-DD |
| `problem_id` | 프로그래머스 문제 번호 |
| `title` | 문제 제목 |
| `level` | 0 또는 1 |
| `started_at` / `solved_at` | HH:MM (로컬) |
| `duration_min` | 총 소요 분 (정수) |
| `first_try_correct` | 첫 제출에서 100점? |
| `saw_solution` | 해설 참고 여부 |
| `stuck_point` | 한 줄 메모 (빈 문자열 가능) |
| `grade` | `Again` / `Hard` / `Good` / `Easy` (주관) |
| `js_equivalent_known` | JS로는 바로 떠올릴 수 있었나? |
| `python_specific_friction` | 파이썬이라서 막힌 지점 (예: "딕셔너리 기본값 처리") |
| `tags` | 자유 태그 (예: `["해시", "구현"]`) |
| `note` | 기타 자유 메모 |

### 등급 기준 (Grade)
- **Again**: 풀었지만 거의 손도 못 댈 뻔했다 / 다시 풀면 또 막힐 듯
- **Hard**: 꽤 고생했다
- **Good**: 무난히 풀었다
- **Easy**: 금방 풀었다

이 등급은 나중 FSRS 적용 시 그대로 입력값으로 씀.

## 학습 단계 전략 (절차적 ↔ 선언적)

**배경**: JS 능숙자는 처음엔 절차적(imperative) 스타일이 자연스러움. 파이썬의 선언적(declarative) 관용구(컴프리헨션, `sum(... for ...)`, `map`, 슬라이싱 등)는 단계적으로 습득. 인지 부하 이론상 **"알고리즘 + 파이썬 문법 + 파이써닉 숙어"**를 동시에 요구하면 과부하.

**원칙**: 절차적으로 먼저 통과 → 선언적 리뷰로 학습 → 점진적 전이.

### Stage A — 절차적 기본값 (현재, ~exp_000)
- **절차적으로 풀기** (for + 임시 변수 OK)
- 통과 후 **선언적 리뷰 받기** (Claude가 대안 제시)
- 선언적 버전은 **이해만** 하고 강제 암기 X

### Stage B — 선언적 본인 작성 시도 (exp_001 이후, Lv.1 안정)
- 처음엔 절차적 유지
- **통과 후 본인이 선언적 버전 직접 작성 시도**
- 본인 작성 vs Claude 제안 비교

### Stage C — 선언적 우선 시도 (Lv.2 진입 즈음)
- **익숙한 유형**: 선언적 먼저 시도 (실패 시 절차적 폴백)
- **새 유형**: 절차적 유지

### Stage 전환 기준

- **A → B**: 같은 파이써닉 패턴을 Claude 리뷰에서 **3번 이상 본** 유형 단위로 전환
  - 예: `sum(... for ...)` 패턴을 3번 봤으면 다음 만날 때 Stage B 시도
- **B → C**: 본인이 쓴 선언적 버전이 Claude 제안과 **절반 이상 일치**하기 시작

### 디버깅 편의 보존

절차적의 장점(**중간 상태 관측**)은 중요한 디버깅 도구. Stage C에서도:
- 복잡한 로직 / 중간 검증 필요한 문제는 **절차적이 합리적**
- 선언적이 항상 정답이 아님 — 상황별 선택 능력이 목표

### FSRS 등급 가이드 (Stage 연계)

- **Easy**: 절차적·선언적 어느 쪽이든 반사적으로 나옴
- **Good**: 통과했고 디버깅 과정이 정상 범위 (중간 print 몇 번)
- **Hard**: 같은 패턴 리뷰에서 **봤음에도** 전이 안 됨 / 타입 실수 등 **선언적이었다면 안 났을 실수**
- **Again**: 로직 자체가 막혀서 해설 참조 직전까지 감

→ Hard를 정직하게 매기면 FSRS가 **Stage B/C 진입 시점을 가속**.
