# ELO 레이팅 시스템

## 1. 배경 / 왜 나왔나

- **Arpad Elo** (헝가리계 미국 물리학자, 체스 마스터)가 1960년대에 제안, 1970년 FIDE(세계체스연맹) 공식 채택.
- 이전 체스 레이팅(Harkness system)은 임시방편 — 승/패만 세고 상대 실력 반영이 거칠었음.
- Elo의 통찰: **실력은 확률 분포로 모델링할 수 있다**. 두 플레이어의 실력 차이가 커질수록 강자가 이길 확률이 체계적으로 높아진다.
- 지금은 체스를 넘어 **Codeforces, AtCoder, 농구/축구 랭킹, 게임 매칭메이킹(초기 형태)** 등 광범위하게 쓰임.

## 2. 핵심 아이디어 (직관)

세 가지만 기억하면 됨:

1. **모든 플레이어에게 숫자 하나(R, 레이팅)를 부여**한다. 초기값 보통 1500.
2. 두 사람이 붙으면 레이팅 차이로 **기대 승률 E**를 계산한다. 레이팅 400 차이 = 10배 승률 차이.
3. 실제 결과(S)와 기대(E)의 **차이만큼** 레이팅을 조정한다. 예상보다 잘했으면 올리고, 못했으면 내린다.

비유: **"너 정도 실력이면 이쯤 이겨야 돼"라는 예상치가 있고, 실제가 그보다 좋으면 상향, 나쁘면 하향**.

## 3. 수식 & 유도

### 3.1 기대 승률

플레이어 A(레이팅 R_A)가 플레이어 B(레이팅 R_B)를 이길 확률:

```
E_A = 1 / (1 + 10^((R_B - R_A) / 400))
```

- R_A = R_B 일 때 → E_A = 1 / (1 + 10⁰) = 1/2 → 50% (대칭)
- R_A - R_B = 400 → E_A = 1 / (1 + 10⁻¹) = 1 / 1.1 ≈ **0.909** → 약 91%
- R_A - R_B = 200 → E_A ≈ **0.76**
- R_A - R_B = -400 → E_A ≈ **0.091**

**왜 하필 10과 400인가?** 임의 선택. Elo 원래 논문은 정규분포 기반이었고, 현재의 로지스틱 + 400 스케일은 USCF(미국체스연맹)가 편의상 정한 관례. 400은 "10배 승률 차이"를 기준점으로 두려는 것.

### 3.2 업데이트 식

경기 후:

```
R_A' = R_A + K × (S_A - E_A)
```

- `S_A` = 실제 결과 (승=1, 무=0.5, 패=0)
- `K` = 변동 계수 (K-factor). 클수록 레이팅이 크게 움직임.
- 체스에서는 보통 **K = 16 (고수), 24 (중수), 32 (신규/저수)**.
- Codeforces는 대회마다 다름, 대략 ±100~200 변동.

### 3.3 제로섬 성질

한 경기에서 A가 얻는 변화량 = B가 잃는 변화량 (K가 같을 때).

```
(S_A - E_A) + (S_B - E_B)
  = S_A + S_B - (E_A + E_B)
  = 1 - 1 = 0
```

(S_A + S_B = 1은 정의상 자명, E_A + E_B = 1은 로지스틱 대칭성.)

→ **레이팅은 닫힌 시스템에서 총합이 보존**된다. 새 유저가 1500으로 들어오면서 총합이 늘어나긴 함.

### 3.4 로지스틱 함수로 다시 쓰기

`10^(x/400) = e^(x × ln10 / 400)` 이므로, `β = ln10 / 400 ≈ 0.00576`.

```
E_A = σ(β × (R_A - R_B))
```

여기서 σ는 시그모이드 `σ(x) = 1/(1+e^(-x))`. 즉 **ELO는 로지스틱 회귀의 특수형**이다. 이 관점이 나중에 베이지안 해석으로 이어짐.

## 4. 예제 계산

**상황**: A=1600, B=1800, K=32, A가 승리.

1. 기대 승률
```
E_A = 1 / (1 + 10^((1800-1600)/400))
    = 1 / (1 + 10^0.5)
    = 1 / (1 + 3.162)
    ≈ 0.240
```
A는 24%만 이길 거라고 기대됐는데 실제로 이김(S_A=1).

2. 업데이트
```
R_A' = 1600 + 32 × (1 - 0.240)
     = 1600 + 32 × 0.760
     = 1600 + 24.3
     ≈ 1624
```

3. B는 대칭:
```
R_B' = 1800 + 32 × (0 - 0.760)
     = 1800 - 24.3
     ≈ 1776
```

**의미**: 약자가 강자를 이기면 ±24점, 강자가 약자를 이기면 ±8점(1 - 0.76 = 0.24 × 32)만 움직임. 업셋이 더 크게 반영.

## 5. 파이썬 최소 구현

```python
def expected(r_a: float, r_b: float) -> float:
    """A가 B를 이길 기대 확률."""
    return 1.0 / (1.0 + 10 ** ((r_b - r_a) / 400))


def update(r_a: float, r_b: float, score_a: float, k: float = 32) -> tuple[float, float]:
    """한 경기 후 두 플레이어 새 레이팅."""
    e_a = expected(r_a, r_b)
    r_a_new = r_a + k * (score_a - e_a)
    r_b_new = r_b + k * ((1 - score_a) - (1 - e_a))
    return r_a_new, r_b_new


if __name__ == "__main__":
    # 1600 vs 1800, A 승리
    new_a, new_b = update(1600, 1800, score_a=1, k=32)
    print(f"A: 1600 -> {new_a:.1f}")
    print(f"B: 1800 -> {new_b:.1f}")
    # A: 1600 -> 1624.3
    # B: 1800 -> 1775.7
```

## 6. 한계 / 언제 안 맞나

1. **불확실성 무시**
   - 3경기 둔 1500 vs 300경기 둔 1500을 같게 취급.
   - 신규 유저는 실력이 덜 파악됐는데도 같은 K로 움직임 → **Glicko가 해결**.

2. **실력 변화 반영 속도**
   - K가 고정되면 급성장/급하락을 빠르게 못 따라감.
   - K를 구간별로 다르게 주는 건 임시방편.

3. **1대1 모델**
   - 팀전/다인전에 바로 못 씀 → **TrueSkill이 해결**.

4. **사람 vs 문제에 부적합**
   - 문제는 "항상 같은 실력으로 응수" → 대결 모델과 성격이 다름.
   - 문제풀이는 **IRT가 더 적합**.

5. **레이팅 인플레이션**
   - 신규 유저가 계속 1500으로 들어오면 전체 평균이 올라감.
   - 대회에 따라 보정식 추가 (Codeforces의 rating inflation 이슈 유명).

## 7. 코딩테스트 학습에 적용 가능한 부분

혼자 공부하는 입장에선 **원형 그대로는 쓸모 없음**(대결 상대가 없음). 하지만 응용:

### 7.1 "나 vs 문제"를 가짜 대결로 모델링
- 문제에 레이팅 b(난이도)를 부여 (solved.ac/Codeforces 난이도 차용)
- 내 레이팅 θ를 두고, 문제를 맞추면 S=1, 틀리면 S=0으로 ELO 업데이트
- **문제 레이팅은 고정**, 내 레이팅만 움직이게 하면 단순 추정기로 작동
- 이 방식이 사실상 **IRT의 1PL 모델과 같다** (뒤에서 이어짐)

### 7.2 유형별 서브 레이팅
- 전체 θ 하나 말고 **DP-θ, 그래프-θ, 그리디-θ** 따로 관리
- 내 약점 유형이 숫자로 드러남

### 7.3 시간 가중 K
- 최근 기록일수록 K 크게 → 최근 실력 변화 빨리 반영
- 오래된 기록은 K 줄여서 노이즈화 방지

### 7.4 주의할 점
- **표본이 적으면 의미 없음** — 한 주에 10문제 풀면 레이팅 변동이 노이즈
- **문제 난이도가 부정확하면 다 틀어짐** — 플랫폼 난이도를 신뢰할 수 있을 때만 의미 있음
